# **The other performance**

Final lap.

AnnieMation is powerful ‚Äî _too_ powerful if unleashed carelessly.

This page is the safeguard: how to keep motion fast, meaningful, and respectful of user experience.

---

## üöÄ Use GPU-Friendly Properties

Stick to transforms + opacity whenever possible.

```css
/* Fast */
transform: translate();
opacity: 0.8;

/* Slow, layout-triggering */
width: 200px;
top: 20px;
left: 30px;
```

Transform + opacity = GPU acceleration.  
Layout-altering properties = reflow city.

---

## üßº Keep Keyframes Lightweight

Avoid animating expensive properties:

- `box-shadow` (heavy)
- `filter` (heavier)
- `blur()` (heaviest)
- `height/width` (layout thrash)
- `border-radius` (surprisingly pricey)

Use sparingly ‚Äî or fake with transforms.

---

## üéö Duration Discipline

Good UI motion is:

- **150‚Äì300ms** for interaction
- **400‚Äì600ms** for entrances
- **700ms+** only for cinematic or storytelling moments

Long animations feel sluggish unless justified.

---

## üìâ Avoid Motion Overload

If _everything_ moves, **nothing** stands out.

Use animation to:

- reinforce hierarchy
- guide attention
- enhance clarity
- support interaction

Not to show off (unless it‚Äôs a demo ‚Äî then go wild).

---

## üßò Respect `prefers-reduced-motion`

Always include it. Always.

```css
@media (prefers-reduced-motion: reduce) {
  * {
    animation-duration: 0ms !important;
    transition-duration: 0ms !important;
  }
}
```

Zero effort, huge accessibility win.

---

## üß© Test on Low-End Devices

Animations that feel perfect on your M‚Äëseries MacBook  
might stutter on lower‚Äëpowered hardware.

If it janks, simplify:

- fewer keyframe steps
- shorter shadows
- fewer simultaneous animations
- reduce blur/filter usage
- reduce DOM complexity in animated scenes
